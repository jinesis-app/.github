# Contributing Guide

Thank you for contributing to this project.

This document explains how to properly create issues, propose features, and submit pull requests.

---

# Issue Guidelines

Before creating a new issue, please check the following:

* The issue has not already been reported
* The issue is related to this project
* You selected the correct issue template

Available issue types:

* **Epic** — Large initiative containing multiple stories
* **Story** — A user-facing feature requirement
* **Feature** — Request for new functionality
* **Improvement** — Enhancement to an existing feature
* **Task** — Technical implementation work
* **Sub-Task** — Small unit of work related to a task
* **Bug** — Unexpected behavior or error

---

# Writing Good Issues

Please provide:

* Clear title
* Detailed description
* Steps to reproduce (for bugs)
* Expected behavior
* Screenshots or logs when applicable

Well-written issues help maintainers resolve problems faster.

---

# Pull Request Guidelines

When submitting a pull request:

1. Create a branch from `develop`
2. Follow the project's coding standards
3. Write clear commit messages
4. Link the related issue

Example:

```
Fixes #123
```

---

# Development Workflow

Typical workflow:

```
Epic
 └── Story
      └── Task
           └── Sub-task
```

Issues should follow this hierarchy whenever possible.

---

# Code Quality

Please ensure:

* Code follows project standards
* Tests pass
* Documentation is updated if needed

---

# Reporting Security Issues

If you discover a security vulnerability, **do not open a public issue**.

Instead, contact the maintainers privately.

---

Thank you for helping improve this project.
